# 案例研究：`task_20_eli5_pdf_summary`

## 任务概览

- 任务 ID：`task_20_eli5_pdf_summary`
- 名称：`ELI5 PDF Summary`
- 来源 trace：[ /hy-tmp/skill/results/rq1/0028_minimax-cn-MiniMax-M2-5.json ](/hy-tmp/skill/results/rq1/0028_minimax-cn-MiniMax-M2-5.json)
- 用户指令：`读取我工作区里的 GPT4.pdf 文件。写一份 ELI5 风格的总结，并保存到 eli5_summary.txt。`

这是一个很有代表性的“首次尝试就产生大量浪费”的案例，因为 agent 仅在第 1 次 attempt 中就消耗了 `1,226,263` 个 token：

- `input_tokens`：`1,056,259`
- `output_tokens`：`3,281`
- `cache_read_tokens`：`166,608`
- `cache_write_tokens`：`115`
- `request_count`：`16`

这里的关键模式并不是“重试成本”。真正的失败更早就发生了：一次对原始 PDF 的直接读取，把巨大的二进制载荷注入了对话上下文，而这个被污染的上下文又被后续 15 轮内部交互持续携带。

## 核心诊断

第一次 attempt 的成本里，实际上混杂着两类完全不同的消耗：

`必要工作`
- 识别 `GPT4.pdf` 不是纯文本。
- 走 PDF 文本提取路径。
- 起草 `eli5_summary.txt`。
- 做一次轻量验证。

`浪费`
- 直接把原始二进制 PDF 读进模型上下文。
- 在犯下这个错误后，带着被污染的上下文进行了多轮补救。
- 基于提取结果写出了事实错误的摘要。
- 花了很多轮编辑去修复模型自己引入的格式和 Unicode 问题。

其中占主导地位的浪费是 `上下文污染（context contamination）`：从第 1 轮之后，后面的每一次模型调用都要携带大约 `64k-73k` 的输入 token。

## 压缩版时间线

### 阶段 1：错误的初始读取污染了上下文

- `Round 1`
  动作：`read GPT4.pdf`
  Token：`10,622 total`（`20 input`）
  评估：意图本身合理，但工具选错了。把二进制 PDF 当成原始文本读取，会造成畸形的转录片段，并把 PDF 字节直接注入上下文。

- 污染证据
  trace 立刻记录了多段解析错误，原始载荷长度分别是 `8,548`、`34,407`、`26,570`、`5,380` 和 `50,392` 个字符。这是第一次重大浪费事件。

### 阶段 2：在膨胀的上下文里补救

- `Round 2`
  动作：检查 `pdftotext`、`pdfgrep`、`strings`
  Token：`75,098 total`（`64,207 input`）
  评估：补救方向本身是必要的，但因为上一步已经把二进制内容塞进了上下文，所以现在每一步都异常昂贵。

- `Round 3`
  动作：`strings GPT4.pdf | head -200`
  Token：`75,480 total`（`64,995 input`）
  评估：低价值探索。它只能说明 PDF 是压缩的，并没有提供真正可用的论文正文内容。

- `Round 4`
  动作：尝试安装 `poppler-utils`
  Token：`77,024 total`（`66,488 input`）
  评估：基本属于浪费。对于一个只需要提取文本的任务来说，安装系统包是一条过重的补救路径。

- `Round 5`
  动作：`pip install pypdf`，然后用 Python 提取文本
  Token：`77,391 total`（`66,848 input`）
  评估：这是有用的一步，也是第一次真正产生有效进展的提取动作。

### 阶段 3：草稿生成了，但建立在错误理解之上

- `Round 6`
  动作：写入 `eli5_summary.txt`
  Token：`81,547 total`（`70,301 input`）
  评估：文件创建本身是必要的，但内容方向已经偏了。

- 这里引入的主要内容错误
  草稿把 PDF 说成是微软研究院的 `"Sparks of Artificial General Intelligence: Early Experiments with GPT-4"`。
  但后面的 validator 指出这是错误的：任务里的 PDF 实际上是 OpenAI 的 GPT-4 Technical Report。
  草稿还声称 GPT-4 能 `"Create pictures from descriptions"`，validator 也指出这并不准确。

这说明：agent 走了一条成本很高的提取路径，但最后仍然生成了事实错误的摘要。所以这些 token 不只是“数量大”，而且“产出低”。

### 阶段 4：自我引发的清理循环

- `Round 7`
  动作：第一次编辑，删除中文字符
  Token：`81,936 total`
  评估：浪费。这是在修复 agent 自己引入的缺陷。

- `Round 8`
  动作：重新读取 `eli5_summary.txt`
  Token：`82,072 total`
  评估：部分属于有用验证，但仍然是在膨胀上下文里进行。

- `Rounds 9-14`
  动作：反复 `edit`，外加一次 `grep -n "超级"` 搜索
  Token：
  `82,529`、`82,662`、`82,861`、`83,065`、`83,161`、`83,441`
  评估：大多属于浪费。这几轮主要是在修复模型自己引入的 Unicode 问题，并恢复误删的小节标题。

- `Round 15`
  动作：`grep -n "�"` 和 `wc -w`
  Token：`83,558 total`
  评估：这一步有用，但出现得太晚了。词数验证本应更早、更便宜地完成。

- `Round 16`
  动作：最终回答
  Token：`83,816 total`
  评估：收尾本身是必要的，但它错误地声称 `458` 词符合要求的 `200-400` 区间。

## 第一次 Attempt 的 Token 都花到哪里去了

### 1. 二进制上下文残留带来的结构性浪费

每一轮的输入 token 为：

`20, 64,207, 64,995, 66,488, 66,848, 70,301, 71,174, 71,596, 71,945, 72,153, 72,286, 72,485, 72,689, 72,828, 73,065, 73,179`

可以这样理解：

- 这个任务本来并不需要一个 `64k+` 的超长 prompt。
- 从 `20` 一下跳到 `64,207`，并且正好发生在原始 `read` 之后，这强烈表明 PDF 字节已经进入了持久对话状态。
- 此后每一轮都在为这次失误重复付费。

保守下界估计：

- 即使我们非常宽松地假设，后续每一轮“本来”真的需要 `10,000` 个输入 token，那么重复产生的超额输入也至少有 `906,239` 个 token。
- 这已经占到第一次 attempt 全部输入 token 的 `85.8%`。

所以，即使先不讨论事实错误和额外编辑，这次 attempt 的主要成本也已经被一次错误的上下文摄入决定了。

### 2. 真正有用的工作

真正有价值的流程其实很短：

1. 识别普通 `read` 不足以处理 PDF。
2. 使用 PDF 文本提取方法。
3. 写出摘要文件。
4. 做一次简短验证。

超出这条最小路径之外的部分，要么是补救成本，要么是清理成本。

### 3. 自我造成的返工

`Round 7-14` 主要都花在修复以下问题：

- 草稿中混入了中文文本。
- 一次过度宽泛的编辑删掉了 `## Are there any problems?` 标题。
- 残留的 Unicode 碎片。

这些都不是用户要求的工作，而是模型自己输出造成的返工。

### 4. 错误内容带来的成本

即使是看起来“有产出”的草稿轮次，也不能算完全必要，因为它总结的是错误的论文，还引入了事实错误。这意味着第 6 轮中也有相当一部分 token 在语义上属于浪费，尽管从 action trace 上看它像是在向前推进。

## 必要成本 vs 浪费成本

仅靠 trace 元数据，无法做到严格精确切分，因为 trace 记录的是每轮总 token，而不是每句话的 token 归因。但从因果关系上看，这个区分依然非常清楚。

`明显必要的`
- 第 5 轮中成功的提取路径
- 第 6 轮初次起草并写文件
- 最后附近的一次轻量验证

`明显浪费的`
- 第 1 轮把原始二进制内容作为上下文摄入策略
- 第 3 轮的 `strings` 探测
- 第 4 轮安装系统包的尝试
- 第 7-14 轮的自我修复循环
- 最终错误声称 `458` 词满足目标长度要求

`局部看似合理，但结构上依然浪费的`
- 第 2、5、8、15 轮：这些动作本身有合理目的，但因为二进制载荷一直残留在上下文里，它们的 token 成本被极度放大了

## 时间线结论

这次首次尝试展示了一种与“agent 重试太多”不同的失败模式：

1. 一次过早的工具选择污染了上下文。
2. 模型继续在这个被污染的上下文里行动。
3. 后续轮次即使局部决策合理，也会变得异常昂贵。
4. 然后 agent 又额外增加了一层浪费，用来修复自己引入的缺陷。

因此，对 RQ1 来说，这个 trace 是一个很强的证据，说明首次尝试里的 token 浪费并不只是“探索成本”。它至少包括：

- `上下文摄入浪费（context-ingestion waste）`
- `补救路径浪费（recovery-path waste）`
- `自我修复浪费（self-repair waste）`
- `错误内容浪费（wrong-content waste）`

## 适用于 RQ1 的简洁表述

如果你想把它写进论文，可以使用下面这句更简洁的表述：

> 在 `task_20_eli5_pdf_summary` 中，首次尝试的大部分成本并不是任务本身所要求的。一次原始 PDF 读取污染了 prompt 状态，之后 agent 在每一轮都反复支付 `64k-73k` 的输入 token。随后，这次尝试又额外花费多轮去修复模型自己引入的输出缺陷，而最终生成的摘要依然存在事实错误。

这使得该 trace 成为一个很好的反例，说明 token 浪费的主要来源并不一定是外部重试。在这里，validator 还没开始反馈之前，浪费就已经非常严重了。

## 关于 Cache 命中的补充说明

从这个 trace 来看，真正命中 cache 的只是一小段稳定前缀，后面真正昂贵的部分基本都是动态上下文，因此几乎不可能继续命中。

先看每轮数字，`task_20` 第一次 attempt 的 `usage_per_round` 里：

- Round 1：`input 20`，`cache_read 10413`
- Round 2：`input 64207`，`cache_read 10413`
- Round 3：`input 64995`，`cache_read 10413`
- ...
- Round 16：`input 73179`，`cache_read 10413`

这里最关键的现象是：

- `cache_read_tokens` 几乎固定不变，始终在 `10413`
- `input_tokens` 却一直维持在 `64k-73k`

这基本说明了两件事：

1. 有一段稳定前缀一直命中了 cache。
2. 后面那一大块上下文每一轮都在变化，所以无法继续命中 cache。

你要的拆分，可以这样理解。

### 更像“稳定前缀，已被 cache”的部分

大概率包括这些：

- 系统提示词
- agent 的固定行为说明
- benchmark task 的原始用户请求
- 一些每轮都不变的工具 schema / 会话固定头部
- 在 provider 侧能识别为“相同前缀”的那部分历史

在这个 trace 里，这部分规模大概就是每轮固定的那 `~10,413` 个 `cache_read_tokens`。

### 更像“动态脏上下文，无法被 cache”的部分

大概率包括这些：

- 第 1 轮 `read GPT4.pdf` 后塞进来的原始 PDF 文本 / 二进制碎片
- 这些碎片导致的 parse error / raw transcript 内容
- 每一轮新的 tool result
- 每一轮新的 thinking / assistant 文本 / tool call
- 对 `eli5_summary.txt` 的逐轮编辑结果和文件内容变化
- 逐轮增长的会话尾部

这部分就是后面每轮 `64k-73k` 输入 token 的主体。

### 为什么后面那么多内容没有命中 cache？

核心原因不是“有 cache 但失效了”，而是这类 prompt caching 通常只对“从开头开始、完全一致的前缀”生效。一旦前面出现了大块脏内容，而后续每轮尾部又不断变化，模型请求虽然“看起来相似”，但并不是“同一个长前缀”。于是会出现：

- 前面一小段固定头部还能命中
- 后面大段 PDF 垃圾、工具结果和编辑历史都是动态的
- 动态部分每一轮都必须重新计算，并记入 `input_tokens`

更直白一点：真正被缓存的，只是那段始终不变的“开头模板”；真正昂贵的，是后面不断变化、而且已经被 PDF 原始内容污染过的长尾上下文。
